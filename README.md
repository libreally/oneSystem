
1. 项目概述
1.1 项目目标
建设一个统一的智能办公平台，由“一系统”和 AI 助手两部分组成。其中：
一系统主要负责对现有业务系统进行统一集成，在不改变原有系统的基础上，通过接口、浏览器或其他方式抓取各系统中的任务、数据、状态和结果，形成统一工作入口。
AI 助手主要负责承载和调度 skills 能力，通过调用已有 skills、自动生成执行逻辑、生成新 skill 等方式完成文档处理、数据处理、流程推进、报告生成等复杂操作，减少每个新需求都依赖开发人员单独开发功能的情况。
1.2 建设目标
平台需具备以下能力：
在不改造原有业务系统的前提下，完成多系统数据和任务的统一集成。通过统一工作台集中展示用户待办、进度、预警、结果和工作总结。
通过 AI 助手调用 skills 完成复杂办公操作，当没有现成 skill 时，由大模型自动生成执行逻辑处理需求。对高频重复需求，支持沉淀为新的标准 skill。
对公文模板、敏感词、流程规则、定时策略等内容提供配置能力。对配置文件进行权限划分，区分公共配置与个人配置。
支持用户个性化分析，静默生成适合用户的使用风格，并在配置中心可查看、可删除。
与本地 WPS 集成，支持直接处理本地 Word、Excel 文件。
支持定时任务、主动提醒、自动报告、自动周报月报等周期性业务。
支持本地文件自动检索与会话知识库构建，不再依赖用户手动上传文件。支持用户自定义本次对话知识库文件。
2. 平台总体定位
2.1 一系统定位
一系统是统一工作台，核心作用是做现有系统集成。在不改变原系统的基础上，对多个系统的数据、任务和状态进行统一抓取、统一展示和统一入口整合。
一系统主要解决的问题：
系统入口分散
用户需要频繁切换多个系统
各系统任务、状态、结果分散
用户缺少统一视角查看工作全貌
2.2 AI 助手定位
AI 助手是平台的智能执行入口，核心作用是承载和调度 skills 能力，帮助用户完成复杂操作。
AI 助手主要解决的问题：
文档处理、数据处理、流程处理等操作复杂
很多需求不值得单独开发一个固定功能
新需求变化快、场景多，传统开发响应慢
用户希望通过自然语言直接完成任务
因此，AI 助手的重点不是简单回答问题，而是通过调用 skills、自动生成逻辑、沉淀新 skill，完成实际业务操作。
3. 平台总体功能
3.1 一系统统一工作台
作为统一入口，主要负责对现有业务系统进行集成和数据聚合，在不改变原系统的前提下，统一展示：
用户待办
任务进度
风险预警
系统消息
自动生成的结果文件和报告
本周、本月工作总结
3.2 AI 自动化助手
负责理解用户自然语言需求，并调度 skills 完成业务操作，包括：
文档处理
数据比对和统计
督办任务提取
流程查询和推进
报告生成
定时任务配置和执行
3.3 Skills 能力体系
将通用能力封装为标准 skill，供 AI 助手调用，减少重复开发。
3.4 Skill 动态生成能力
当现有 skill 不满足需求时，大模型可根据用户需求生成临时执行逻辑；当该需求高频复用时，可继续沉淀为新的标准 skill。
3.5 配置中心
统一维护公文模板、敏感词库、督办规则、推荐规则、定时策略、公共配置、个人配置等内容。
3.6 定时调度中心
支持按时间、周期或条件自动执行任务，实现主动提醒、自动抓取、自动报告和自动汇总。
3.7 本地 WPS 集成
支持 AI 助手和相关 skill 直接处理本地 Word、Excel 文档。
3.8 本地文件自动检索与会话知识库
支持根据当前提问内容，自动在用户电脑中检索相关文件，构建本次对话的临时知识库，并支持用户手动指定本次对话使用的文件。
4. 一系统功能需求
4.1 集成目标
在不改变原有平台系统等业务系统的前提下，将分散的数据和任务统一抓取并集中展示。
4.2 主要功能
4.2.1 任务聚合
从多个业务系统抓取并展示：
今日待办
近期计划
即将超期任务
高优先级任务
风险提醒
4.2.2 智能推荐
系统应根据时间、任务上下文、角色和配置规则，动态推荐可执行操作，服务卡片。
4.2.3 结果中心
统一展示：
系统执行结果
生成文档
生成报表
定时任务结果
失败告警
4.2.4 工作总结板块
自动汇总用户本周、本月工作情况，展示：
已完成事项
未完成事项
当前推进中的重点任务
风险和超期事项
已生成成果文件
下周、下月工作建议
支持可编辑操作，自动保存已编辑内容
4.2.5 用户画像
结合用户以往对话记录进行分析，包括：
推荐更符合用户习惯的服务入口
优先展示用户常用任务类型
优化总结内容的展示方式
提前推荐用户高频使用的模板或操作
调整 AI 助手默认输出风格
该能力后台系统静默生效，不主动打扰用户，用户仅可在配置中心查看、管理或删除。
4.3 集成原则
不改变原有业务系统
不影响原有系统正常使用
优先通过接口方式抓取数据
无接口时通过浏览器或 其他方式进行数据抓取集成
用户前端尽量只看“一系统”，后台完成系统间调用
5. AI 助手功能需求
5.1 自然语言理解与参数识别
用户可直接输入业务指令，例如：
把这个文件转成标准公文
检查这篇材料的敏感词
合并这两个表
找出没考试的人
提取会议中的督办任务
查询任务 A 当前状态
将任务 B 推进到下一阶段
每周五推送下周作业计划
系统需自动识别：
用户意图
输入对象
文件路径或上传文件
业务类型
时间范围
模板和规则要求
输出方式
5.2 缺失参数补充机制
当用户输入的信息不足以直接执行任务时，系统不能直接执行，应先判断缺失的关键参数，并向用户发起补充询问，待信息补齐后再继续执行。
典型情况包括：
未提供文件或文件路径
未说明处理对象是当前文档、上传文件还是本地文件
未指定模板、公文类型或规则配置
未说明匹配字段、sheet 名称或统计范围
未说明时间范围、推送周期或执行时间
未说明输出格式或输出方式
用户表达存在歧义，无法确定具体执行目标
要求如下：
仅在关键参数齐全后执行任务
可使用默认值的参数，可提示用户是否使用默认配置
无法确认的关键参数，必须先询问用户
用户补充信息后，系统应结合上下文继续执行，无需用户重复描述完整需求
5.3 三级执行机制
5.3.1 调用已有 skill
优先调用已存在的标准 skill 完成任务。
5.3.2 自动生成执行逻辑
若没有现成 skill，但可以通过脚本、代码解释器或多个能力组合完成，则由大模型自动生成并执行。
5.3.3 生成新 skill
若某类需求高频出现，平台应支持将其沉淀为新 skill，纳入 skill 库。
5.4 输出能力
AI 助手执行后应支持：
对话中展示结果摘要
生成 Word、Excel、PDF 文件
返回文件路径
自动打开结果文件
推送到消息中心
触发后续流程或定时任务
5.5 用户个性化分析能力
系统应具备用户个性化分析能力，能够基于用户历史对话内容、使用习惯和操作偏好，自动生成适合用户的助手使用风格和默认行为策略。
5.5.1 分析内容
系统可自动分析并形成以下个性化内容，例如下：
常用任务类型
常用输出格式偏好
常用文档处理风格
常用数据分析方式
常用提醒和推送习惯
常用表达方式和交互偏好
常用模板和规则选择倾向
5.5.2 使用方式
该能力应采用静默方式运行，要求如下：
系统自动分析用户已有对话和使用记录
不在日常使用过程中主动打扰用户
不单独弹窗提示用户进行确认
自动将分析结果用于优化助手响应风格和默认处理方式
仅在配置中心中提供查看、管理和删除入口
5.5.3 用户可控要求
系统需支持：
在配置中心查看个性化分析结果
删除已有个性化画像或偏好记录
关闭或停用个性化能力
对个人偏好进行手动修改
5.5.4 边界要求
个性化分析仅用于优化用户体验和平台执行效果，不应影响公共配置的统一规则，也不应突破权限边界。涉及公共模板、流程规则、组织级标准时，仍应以公共配置为准。
6. Skills 管理与生成功能
6.1 Skill 管理
每个 skill 需要具备以下信息：
skill 名称
功能说明
适用场景
输入参数
输出结果
调用方式
依赖配置
权限要求
版本信息
日志记录
6.2 Skill 生成能力
平台需要提供“生成 skill 的能力”，支持：
根据需求描述生成 skill 模板
生成输入输出定义
生成执行逻辑
关联所需配置项
注册到 skill 管理中心
支持测试、审核、发布、停用和版本管理

7.1 公文模板配置
支持配置：
公文类型
标题格式
正文字体、字号、行距
页边距
段落缩进
页码格式
落款格式
红头
发文号
签发人
附件说明
是否保留原文结构
是否自动生成目录
7.2 敏感词库配置
支持配置：
敏感词导入
分类管理
分级管理
正则表达式匹配
删除、替换、标记等处理方式
不同文档级别使用不同词库
是否处理页眉页脚、表格等内容
7.3 督办规则配置
支持配置：
状态机
合法流转路径
临期和超期规则
提醒规则
权限规则

7.4 智能推荐规则配置
支持配置：
触发条件
推荐对象
时间规则
关键词规则
展示文案
触发动作
7.5 定时任务配置
支持配置：
执行周期
执行时间
触发条件
节假日规则
重试策略
输出方式
推送渠道
7.6 配置文件权限划分
配置中心中的配置文件需支持分级管理和权限划分，不同类型配置应具有不同的可见范围和可编辑权限。
7.6.1 配置分类
1. 公共配置。由平台管理员或具备权限的管理人员统一维护，用于全平台或指定组织范围内统一生效的配置。普通用户可查看可用范围内的公共配置，但不得修改。适用内容包括：
公文标准模板
公共敏感词库
督办流程规则
定时任务公共策略
智能推荐公共规则
组织级通用配置
2. 个人配置。由用户自行维护，仅对用户本人或其授权范围内生效。用户可自行查看、修改、启用、停用或删除。适用内容包括：
个人常用模板
个人输出偏好
个人提醒偏好
个人工作总结偏好
个人常用分析规则
个人助手使用习惯配置
7.6.2 权限要求
配置文件需支持以下权限控制能力：
区分查看权限与编辑权限
区分公共配置与个人配置的管理边界
普通用户不得修改公共配置
管理员可维护公共配置
用户可维护个人配置
支持按组织、角色、部门划分配置使用范围
支持配置版本管理和变更记录
7.6.3 使用要求
系统在执行任务时，应优先按以下顺序读取配置，示例如下：
1.用户个人配置
2.用户所属范围内可用的组织级配置
3.平台默认公共配置
4.若用户未设置个人配置，则自动使用公共配置或默认配置。
8. 本地文件自动检索与会话知识库构建
8.1 功能目标
当前平台知识库主要依赖用户手动上传文件。为增强平台与用户电脑的深度集成能力，系统需支持在用户发起问题后，自动根据当前提问内容，在用户本地电脑中检索相关文件，并自动构建本次对话所需的临时知识库，用于辅助回答问题和执行任务。
该能力的目标是：
不再要求用户每次手动上传文件 
根据当前问题自动发现相关资料 
不仅依赖文件名检索，还要基于文件内容进行检索 
将命中的相关文件自动纳入当前会话知识范围 
支持用户手动指定、增删本次会话使用的知识库文件 
8.2 功能范围
8.2.1 本地文件自动检索
当用户提出问题或任务时，系统应能够基于当前对话内容自动分析检索意图，并在用户本地电脑指定范围内检索相关文件。
检索范围可包括但不限于：
用户常用工作目录 
文档目录 
下载目录 
桌面目录 
用户授权的其他本地目录 
8.2.2 多维度检索能力
系统检索文件时，不应仅依赖文件名称，还应支持以下方式：
文件名匹配 
文件内容语义匹配 
关键词匹配 
文档标题匹配 
文档正文段落匹配 
表格内容匹配 
历史使用频次或近期访问记录辅助排序 
也就是说，系统应支持“基于内容的检索”，而不是只做文件名搜索。
8.2.3 会话级临时知识库构建
系统在检索到相关文件后，应自动将本次命中的文件构建为当前会话的临时知识库，用于：
回答用户问题 
提供文件内容摘要 
进行多文件综合分析 
支持后续连续追问 
支持引用相关内容作为回答依据 
该知识库应以“当前会话”为边界，不强制永久入库。
8.3 处理流程要求
系统处理流程应包括：
用户提出问题或任务 
系统分析当前问题意图和可能涉及的资料范围 
在用户授权的本地目录中自动检索相关文件 
根据文件名、内容、语义相关度对候选文件排序 
选取与当前问题最相关的文件构建本次会话知识库 
基于本次会话知识库回答问题或执行任务 
在后续对话中持续复用本次会话知识库，直到会话结束或用户主动调整 
8.4 用户可控功能
除了自动检索，系统还需支持用户对“本次对话知识库”进行自定义管理。
8.4.1 用户可查看本次会话知识库文件
系统应支持向用户展示当前会话已纳入知识库的文件清单，包括：
文件名称 
文件路径 
文件类型 
命中原因或相关性说明 
是否已加载到本次会话知识库 
8.4.2 用户可自定义本次会话知识库文件
用户应支持对本次对话知识库进行手动干预，包括：
手动添加文件到本次会话知识库 
从本次会话知识库中移除文件 
指定仅使用某几个文件回答问题 
清空当前会话知识库并重新检索 
锁定某些文件作为本次对话固定知识来源 
8.4.3 用户优先级要求
当用户明确指定本次对话使用哪些文件时，系统应优先使用用户指定文件，不再盲目扩大检索范围。
8.5 文件类型支持要求
系统应至少支持以下文件类型的检索与解析：
doc 
docx 
txt 
pdf 
xls 
xlsx 
csv 
ppt 
pptx 
后续可扩展支持：
wps 
图片类文档中的文字内容 
markdown 
json 
xml 
日志文件 
8.6 检索与构建要求
8.6.1 检索要求
支持按问题内容自动触发检索 
支持模糊匹配和语义匹配 
支持多文件联合命中 
支持检索结果相关度排序 
支持限定目录、限定文件类型、限定时间范围检索 
8.6.2 知识库构建要求
构建的是本次会话临时知识库 
知识库内容可随会话动态更新 
后续追问可继续使用当前会话知识库 
用户可追加或删除知识文件 
会话结束后可按策略清理或转存 
8.6.3 回答要求
系统在基于本地文件回答问题时，应尽量说明：
参考了哪些文件 
主要依据哪些内容 
是否存在文件内容冲突 
是否需要用户进一步指定资料范围 
8.7 与 AI 助手的协同要求
该能力应作为 AI 助手的基础能力之一，接入到 AI 助手的任务处理链路中。
即当用户提出问题时，AI 助手不只是判断调用哪个 skill，还应先判断：
当前问题是否需要本地文件支持 
是否需要自动检索电脑中的相关资料 
是否需要先构建本次会话知识库再回答 
对于依赖本地资料的问题，AI 助手应优先执行“自动检索本地文件 → 构建会话知识库 → 回答问题/执行任务”的流程。
8.8 与配置中心的关系
该能力建议在配置中心中支持以下配置项：
默认检索目录范围 
是否允许自动检索本地文件 
允许检索的文件类型 
单次会话最多纳入的文件数量 
会话知识库保留策略 
是否优先检索最近使用文件 
是否允许用户手动调整会话知识库文件
9. 本地 WPS 集成功能
平台需支持与本地 WPS 深度集成，至少包括：
识别当前 WPS 活动文档
打开指定本地文件
处理 Word、Excel 文件
生成新的结果文件
返回生成路径
可选自动打开结果文件
默认不覆盖原文件
尽量后台静默执行，不影响用户当前操作
适用场景包括：
公文转换
敏感词清洗
Excel 统计与合并
会议纪要整理
报告生成

需要对以下能力做权限控制：
流程推进
敏感数据访问
定时任务创建
新 skill 发布
自动代码执行
公共配置维护

系统需记录以下内容的变更和管理日志：
公共配置的新增、修改、删除记录
个人配置的新增、修改、删除记录
个性化分析结果的生成时间和更新记录
用户对个性化内容的删除、关闭、修改记录
其中，公共配置变更需保留操作人、时间、变更前后内容和影响范围。
11. 总结
本项目要建设的是一个 统一入口 + AI 助手 + skills + 配置中心 + 定时调度 + 本地 WPS 集成 的智能办公平台。其中：
一系统负责在不改变原有系统的基础上，完成多系统数据抓取、任务聚合、统一展示以及工作总结展示。
AI 助手负责集成和调度 skills，完成复杂业务操作，并通过动态生成能力减少每个需求都依赖开发人员单独开发功能的情况。

用户自然语言提出需求

没有现成 skill 时自动生成执行逻辑
高复用新需求进一步沉淀为新 skill
公共规则和个人配置分级管理
周期性业务通过定时调度中心自动执行
